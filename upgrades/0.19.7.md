## What Changed

### Automatic Global Install Cleanup

Stale global instar binaries have been a persistent source of version confusion. Agents would report old versions because CLI commands resolved to a global binary instead of the shadow install. Multiple global installs could lurk across different Node.js version managers (asdf, nvm, volta) and system-wide locations without anyone noticing.

**What's fixed**: The server now automatically detects and removes stale global installs at startup and after every successful auto-update. Covers asdf, nvm, homebrew, volta, and system-wide install locations. The boot wrapper no longer falls back to a global binary — if the shadow install is missing, it fails loudly with recovery instructions.

### Fixed: /new Topic Sessions Dying Immediately

When a user ran /new to create a topic, the handler spawned a session with a meta-message ("New session started"). Claude would respond and immediately exit — there was no real work to do. When the user's actual message arrived moments later, the session was dead but the server still reported "Delivered".

**What's fixed**: The /new command now only creates the Telegram topic without spawning a session. The first real user message triggers session creation through the normal auto-spawn routing, ensuring the session has actual content to process and stays alive.

## What to Tell Your User

- **Version reporting is now reliable**: Your agent will always report its actual running version. Stale global installs that caused wrong version numbers are automatically cleaned up.
- **New topics work properly**: When you create a new topic, send your message and the session will start up with your content. No more "Delivered" messages that go nowhere.
